- AI 에이전트가 **무엇이고 무엇이 아닌지** 구분할 수 있다

- 에이전트를 **기획 → 발굴 → 해부 → 조립 → 검증 → 출하**하는 전 과정을 직접 수행할 수 있다

# Part A — 이해

## 1. AI 에이전트란 무엇인가

**AI 에이전트 = 목표를 향해 인지(perceive) → 추론(reason) → 행동(act)을, 매 단계 인간 승인 없이 자율 반복하는 시스템.**

- 한 줄 비유: *"대화하는 AI"가 아니라 "일을 끝내는 AI"*
- 동작 루프: **plan(계획) → act(행동) → observe(관찰) → adapt(조정)** — 작업 완료까지 반복
- 단순 지시 수행을 넘어 **스스로 하위 목표를 세우고, 도구를 써서 외부 세계를 바꾼다**

### 챗봇과 무엇이 다른가
- **챗봇**: 1문 1답. 말로 답하고 끝.
- **에이전트**: 말을 *수단*으로 삼아 도구(검색·코드·API·파일)를 호출해 **실제 결과물**을 만든다.

### 워크플로우와 무엇이 다른가 (중요)
- **워크플로우**: 사람이 단계를 미리 정하고, 모델이 그 칸을 채운다. 절차 고정.
- **에이전트**: 환경 + 호출 가능한 도구 + 목표를 정의한 시스템 프롬프트, 그 자체. **절차를 스스로 구성**한다.

> 핵심: 에이전트의 본질은 "자율 루프 + 도구 사용 + 목표 지향". 이 셋이 없으면 에이전트가 아니다.

## 2. 핵심 구성 요소

에이전트는 **5대 요소**로 이루어진다 — 인지·추론·메모리·계획·도구.

| 요소 | 하는 일 |
|------|---------|
| **인지** (Perception) | 텍스트·이미지·로그·이벤트를 읽어 현재 상태와 의도를 파악 |
| **추론** (Reasoning) | LLM으로 상황 이해, 선택지 평가, 전략 수립, 자기 반성 |
| **메모리** (Memory) | 단기(컨텍스트) / 장기(지식·RAG·벡터DB) / 작업(중간 산출물). **2026년 1급 요소** |
| **계획** (Planning) | 목표를 작은 태스크로 분해, 실행 순서 결정 |
| **도구·행동** (Tools/Action) | API·코드 실행·웹·파일 호출 + 실행 후 결과 평가 |

여기에 **RAG**(학습 데이터 밖 외부 지식의 동적 검색)가 더해진다.

> 학습 포인트: 메모리는 2026년 들어 1급 아키텍처 요소가 됐다. 3계층 — core(항상 인컨텍스트)·archival(외부 벡터스토어)·recall(대화 이력).

## 3. 아키텍처와 설계 패턴

### 단일 에이전트의 내부 4-레이어
```
인터페이스(사용자)  →  오케스트레이션(플래너·제어흐름)  →  작업 에이전트  →  인프라(도구·데이터)
```
(아래 "조직 규모 배포 아키텍처"의 4-Layer 포털과는 별개 — 이쪽은 에이전트 한 개의 내부 구조다.)

### 대표 설계 패턴
| 패턴 | 내용 |
|------|------|
| **ReAct** | Reasoning(생각) ↔ Acting(행동)을 교대 반복, 매 행동 후 결과 관찰 |
| **워크플로우 5패턴** | prompt chaining · routing · parallelization · orchestrator-workers · evaluator-optimizer |
| **계층적 오케스트레이션** | 전략 상위 에이전트 + 기능 하위 에이전트 협업 |
| **하이브리드** | 자율 행동 위에 정책·가드레일을 씌워 예측 가능성 확보 |

### 단일 vs 멀티 에이전트
- **단일**: 단순·책임 명확·빠름·저비용. 복잡 업무엔 한계.
- **멀티**: 역할 분담(Planner·Executor·Critic·Auditor), 병렬, 확장성. 비용·조율 복잡성 증가.
- 2026 추세: *"강한 에이전트 1 + 보조 전문 에이전트들"* + 그래프 기반 오케스트레이션.

### 조직 규모 배포 아키텍처 — 4-Layer 에이전트 포털

에이전트가 많아지면 개별로 두지 않고 **포털(Agent Portal)**로 묶어 배포·운영한다. 업계 메인스트림 패턴 — "Internal AI Agent Portal / Agent Marketplace 모델" (MS Copilot Studio·Salesforce Agentforce 등).

| 레이어 | 역할 |
|--------|------|
| **L1 사용자 포털** | 업무별 에이전트 카탈로그·검색·즐겨찾기 — 직원 채택률의 90%를 좌우 |
| **L2 오케스트레이터** | 단일 vs 복합 업무 판단, 에이전트 간 협업 조율 ("에이전트를 부르는 에이전트") |
| **L3 모델 라우터** | 4 AI(Claude·GPT·Gemini·Grok) 중 업무별 자동 선택 — 비용·성능 최적화 |
| **L4 데이터·보안 게이트웨이** | 사내 데이터 격리·익명화·감사 로그(RAG) — 데이터가 외부 API로 새지 않도록 |

**멀티 LLM 모델 선택** — 한 모델로는 한계가 있어 업무 특성별로 고른다 (아래는 *일반적 강점 경향*이며 절대 기준은 아니다 — 모델은 빠르게 바뀐다):

| 용도 | 적합 모델 |
|------|----------|
| 코드·추론 | Claude |
| 한국어 비즈니스 문서 | GPT |
| 실시간 정보 | Grok |
| 멀티모달·이미지 | Gemini |

> 핵심 통찰: **강력한 모델 + 나쁜 아키텍처 = 일관된 실패 / 평범한 모델 + 좋은 아키텍처 = 프로덕션급.** 아키텍처가 모델 성능보다 결과를 좌우한다.

## 4. 에이전트 유형 분류와 구현 형태

### 4-A. 개념적 유형 분류 (여러 축으로 교차 분류)
| 분류 축 | 유형 |
|---------|------|
| 자율성 | 보조형 → 반자율형 → 완전자율형 (코파일럿→코워커→오토파일럿) |
| 역할·기능 | 정보검색형·실행형·의사결정형·협업형·코딩형 |
| 구조(고전) | 단순반사·모델기반·목표기반·유틸리티기반·학습 에이전트 |
| 에이전트 수 | 단일 ↔ 멀티(MAS) |
| 엔터프라이즈 6유형 | Customer·Employee·Creative·Code·Data·Security |

### 4-B. 구현 형태 분류 — "어떤 그릇에 담는가" (⚠️ "에이전트=스킬"이 아니다)
| 그룹 | 형태 | 호출 |
|------|------|------|
| Claude Code 내부 | 스킬(SKILL.md) / 서브에이전트(.md) / 자동 트리거(hook) | `/스킬` · `Task()` · 이벤트 |
| 독립 소프트웨어 | CLI 도구·스크립트 / 웹앱·사이트 내장 / API 서비스 / MCP 서버 | 터미널·화면·HTTP |
| 플랫폼·시스템 | Managed Agent / 데스크톱·OS / 임베디드·로봇 | 플랫폼·디바이스 |

> 같은 에이전트라도 구현 형태는 여럿. 스킬은 형태 중 하나일 뿐이다.

## 5. 에이전트 vs 유사 개념

**"자동화 = 에이전트"가 아니다.** 경계는 *"절차가 고정인가 vs 목표를 받아 절차를 스스로 구성하는가"*.

| 개념 | 동작 | 적응성 | 에이전트와의 차이 |
|------|------|--------|------------------|
| 스크립트 | 정적 코드 실행 | 없음 | 환경 인지·판단 없음 |
| RPA | UI 규칙 자동화 | 규칙 벗어나면 실패 | 의미 이해·의사결정 없음 |
| 워크플로우 자동화 | 사전 정의 흐름 | 분기 로직만 | 흐름을 동적 재구성 못 함 |
| 챗봇 | 대화 응답 | 대화 한정 | 멀티스텝 실행·도구 제한적 |
| **AI 에이전트** | **목표 중심 동적 계획·도구·자기수정** | 높음 | **상황 따라 절차 자체를 재구성** |

> 한 줄: 자동화·RPA·스크립트는 "정해진 절차 실행", 에이전트는 "목표를 받고 절차를 만든다".

# Part B — 제작

## 6. 에이전트 제작 — 전체 프로세스 한눈에

에이전트 제작은 **"제조 공장"**이다 — 표준 단계는 **기획 → 발굴 → 해부 → 조립 → 검증 → 출하**.

```
기획 (7항) → ⛔PO 승인 → 발굴·해부 (8항) → 조립 (8항) → 검증·QC (9항) → 자산화·출하 (9항)
```

3대 원칙:
1. **막연한 요청을 그대로 제조에 넣지 않는다** — 충분히 기획부터 (7항)
2. **남이 만든 것을 통째로 신뢰하지 않는다** — 해부해서 건강한 부품만 (8항)
3. **만든 사람이 자기 검증하지 않는다** — 별도 검증 (9항)

---

## 7. 1단계: 요구 발굴·기획 (스무고개)

**막연한 한 줄 요청 → 충실한 기획안.** AI가 묻고 사람이 답하는 **스무고개**로 진짜 스펙을 끌어낸다.

- 20라운드 4구간: ① 좌표 잡기 → ② 가설·제약 검증 → ③ 차별점·심화 → ④ 기획 종합
- 목적은 "한 문장 압축"이 아니라 **충분한 기획**. 부족하면 20을 넘겨 더 묻는다.

**구간별 질문 예시** (AI가 PO에게 이렇게 묻는다):
- ① 좌표 — "없애려는 수작업이 구체적으로 무엇입니까?" / "언제 발동합니까 — 사람 호출? 이벤트?"
- ② 가설·제약 — "어떻게 잘못될 수 있습니까(실패 모드)?" / "지금 이 일을 누가·어떤 도구로 합니까?"
- ③ 차별점 — "비슷한 기존 도구와 무엇이 달라야 합니까?" / "반드시 직접 만들 부분은?"
- ④ 종합 — "기획안 10항목 중 빈 칸은?" / "이대로 만들면 무엇이 빠집니까?"

### 산출물 1 — 에이전트 기획안 (10항목)
목적·Pain Point / 트리거 / 입력·출력 / 제약조건 / 실패 모드 / 사용자 / 차별점 / 아키텍처 방향 / 필요 부품 / 구동 방식.

### 산출물 2 — 아키텍처 SVG
대상 에이전트의 **관계도 + 흐름도**를 다이어그램으로. 텍스트 기획안이 못 잡는 구조적 누락(끊긴 흐름·고아 부품·순환 의존)이 그림에서 드러난다.

> 그다음: 기획안을 **MBO 목표서**로 만들어 PO 승인을 받는다. 승인 없이 제조 착수 금지.

## 8. 2단계: 발굴·해부·조립

### 8-1. 발굴 — 이미 잘 만든 것을 찾는다
공개 저장소·OSS·Hugging Face·MCP에서 후보 5~10개 수집. **GitHub Stars 등 검증 신호가 명확한 것만** 후보로.

### 8-2. 해부·검증 — "장기 이식" (제작의 심장)
발굴물을 **통째로 쓰지 않는다.** 부품(함수·모듈·프롬프트) 단위로 해부해 분류:

| 판정 | 의미 | 처리 |
|------|------|------|
| 🟢 건강 | 검증 통과·충돌 없음 | 그대로 이식 |
| 🟡 재단 필요 | 골격 양호, 수정 필요 | 재단 후 이식 |
| 🔴 병변 | 보안 위반·deprecated·충돌 | **도려낸다 (이식 금지)** |
| 🏭 자체 제작 | 건강한 대체재 없음 | 신규 작성 |

→ 건강한 부품만 모은 목록 = **BOM(부품 창고)**.

### 8-3. 조립 — 8가지 구동 방식

이 공장은 **LLM(두뇌) 의존형 에이전트** 한 종(種)만 제조한다 — AI 플랫폼 종속형 서브에이전트·스킬·훅은 제조 대상이 아니다(스킬 제조는 형제 공장 `/skill-create` 담당). 단, 그 에이전트가 *어디서 어떻게 구동되는지*는 8가지다 — 운영 루프에 사람이 개입하느냐로 갈린다. 요구에 맞는 구동 방식을 판별해 BOM 부품 + 자체 코드로 조립한다.

**사람 개입형 (5종)** — 운영 루프에 사람이 개입(시키고·보고·받음)

| 구동 방식 | 제작 요점 |
|----------|----------|
| 웹사이트형(웹앱) | 서버 호스팅 + 브라우저 UI · 검색 비노출 필요 시 noindex |
| 데스크톱·로컬앱형 | 로컬 실행 스크립트/앱 · **데모 모드 폴백 필수**(키 없어도 샘플로 동작) |
| 모바일앱형 | 모바일 UI/앱 패키징 |
| 메신저봇형 | Slack·카톡·Teams 봇 연동(웹훅·봇 API) |
| 확장·애드인형 | 브라우저 확장 / Notion·Excel 애드인 패키징 |

**무인 자율형 (3종)** — 사람 개입 없이 자율 구동

| 구동 방식 | 제작 요점 |
|----------|----------|
| 서버상주(데몬)형 | 서버 24/7 상주 + 가드레일(오작동 시 fail-open) |
| API엔드포인트형 | API 서버로 노출 + 인증·rate limit |
| 스케줄·트리거형 | cron·webhook 등록 + 실패 시 안전(fail-safe) |

조립되는 에이전트는 반드시 **5요소**를 갖춘다 — LLM(두뇌)·자율 루프·메모리·도구·목표.

**조립 원칙**: 🟢 건강 부품은 출처 주석 달고 그대로 이식 / 🟡 재단 부품은 톤 맞춰 수정 후 이식 / 🏭 자체 부품은 신규 작성 / 🔴 병변 자리는 자체 제작으로 메운다 (도려낸 것 되살리기 금지).

## 9. 3단계: 검증·QC·출하

> 여기서 **검증 = QC** — 출하 전 품질 확인을 가리키는 같은 말이다.

### 자기검증 금지
만든 사람이 자기 에이전트를 검증하지 않는다. **별도 Verification 주체**가 검증한다.

### 규모별 차등 QC
| 규모 | QC |
|------|-----|
| S (작음) | review-evaluate |
| M (중간) | + 다관점 토론 |
| L (배포형·UI) | + 심층 진단·감사 |

- **"curl 200 ≠ 동작함"** — HTTP 200·파일 생성은 데이터 KPI일 뿐. 실제 호출·사용자 여정으로 검증.
- 완료 기준(코드): ① 파일 존재 ② 빌드 통과 ③ **실제로 호출·실행됨** — ③이 핵심.

### 자산화·출하
만들고 끝이 아니다. 구동 방식에 맞게 등재 + 자산 목록에 기록까지 해야 **출하 완료**.

## 10. 실전 — 제작 체크리스트·사례·교훈

### 제작 체크리스트
```
[ ] 스무고개로 충분히 기획 → 기획안 10항목 + 아키텍처 SVG
[ ] MBO 목표서 → PO 승인 (승인 없이 제조 금지)
[ ] 발굴 — 검증된 후보 수집
[ ] 해부 → BOM (🟢🟡🔴🏭 분류, 병변은 도려냄)
[ ] 구동 방식 결정 → 조립
[ ] 별도 검증자 QC (자기검증 금지, curl 200 ≠ 동작)
[ ] 자산화·등재 → 출하
```

### 실전 사례 — KD엔지니어링 사내 AI 에이전트 9칸
엔지니어링 회사의 3도메인(관리·입찰·엔지) × 3 Pain Point = **9개 에이전트**를 한 칸씩 입주.
- 검증된 OSS만 사용 (Docling ★37k · YOLOv8 ★35k 등 — Stars 명시)
- 일관된 README 6단 구조 / 데모 모드 폴백(키 없어도 동작) / 인간 검증 게이트("설계 확정은 기술사")
- 빅뱅 도입이 아니라 **고통 큰 칸부터 한 칸씩** 점진 입주

### 흔한 실패와 교훈
| 실패 | 교훈 |
|------|------|
| 막연한 요청을 그대로 제작 | 스무고개로 충분히 기획부터 |
| 발굴물을 통째로 채택 | 해부 → 건강한 부품만 이식 |
| 만든 사람이 자기 검증 | 별도 검증자 필수 |
| curl 200 보고 "완료" | 실제 클릭·호출로 검증 |
| 만들고 등재 안 함 | 등재까지가 완료 — 안 그러면 다음에 또 만든다 |

### 조직에 에이전트를 도입할 때 — 5가지 함정

에이전트 1개가 아니라 조직 전체에 에이전트를 깔 때 빠지는 함정:

| 함정 | 해법 |
|------|------|
| ① 업무별 에이전트 폭발 (수백 개 → 어디서 뭘 쓸지 모름) | 영역×단계 매트릭스로 1차 분류 + 계층 구조 |
| ② 단일 에이전트로 안 끝남 (복합 업무) | 오케스트레이터 — "에이전트를 부르는 에이전트" 계층 |
| ③ 각 에이전트가 API 직접 호출 → 비용 폭발 | 모델 라우터 — 업무별 적정 모델 자동 라우팅 |
| ④ 사내 데이터가 외부 API로 유출 | RAG 게이트웨이 — 익명화된 컨텍스트만 전송 |
| ⑤ "내가 다 만든다"는 병목 | Self-Made Skills — 직원이 자기 업무 에이전트를 직접 등록 |

> ⚠️ **Adoption이 진짜 관문**: AX 실패의 약 **70%는 기술이 아니라 "직원들이 안 쓴다"**는 문제다. "내 업무 = 내 에이전트"가 명확해야 채택된다. 잘 만드는 것보다 **쓰이게 만드는 것**이 더 어렵다.
